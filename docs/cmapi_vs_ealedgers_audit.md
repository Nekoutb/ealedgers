## §4 — Where they diverge (gaps)

**Blocking (must reconcile before mirroring is meaningful):**

1. **No artefact-based deploy.** ealedgers runs pip on the prod box; cmapi only ships pre-built tarballs. This is the load-bearing difference — without it, none of items 2-5 are reachable.
2. **No atomic releases / rollback.** ealedgers has one live checkout; a bad deploy is a `git reset` away from a recovery, not a symlink swap.
3. **No bootstrap script.** ealedgers' server was hand-provisioned. There is no `vps-bootstrap.sh` equivalent, so the host is non-reproducible.
4. **No CI gate.** ealedgers has zero pre-merge automated checks — tests run only locally and get reported in the PR body. Any of the 99 tests could regress silently between merges.
5. **Static SQLite database not yet cut over.** PR #21 made the code Postgres-ready but the live data is still SQLite. cmapi-mirror parity requires Postgres on the box.

**Cosmetic (defer until §5 items 1-5 land):**

6. SSH password vs SSH key — ealedgers' key-based auth is actually safer; do **not** downgrade to match cmapi.
7. nginx vs Apache — Apache is co-tenant-load-bearing for the other two vhosts on the box; switching is high-risk for no win.
8. Trunk-based vs PR-based — ealedgers' PR discipline is healthier than cmapi's direct-push; keep the PRs.
9. Slice-numbered commits — nice-to-have, not load-bearing.
10. Plan-document depth (WBS, multiple plan files, TEST GATE protocol) — partially adopted via `docs/EXECUTION_PLAN.md`; expand as needed.

## §5 — Concrete recommendations (PR-sized, sequential)

### Step 3 — Finish the Postgres cutover

What changes: Execute the runbook in `docs/postgres-cutover.md` on the live box. Switch `DATABASE_URL` in `/opt/ealedgers/.env` to the new local PG, run `migrate`, dump-and-load the SQLite data, restart gunicorn, smoke-test.

Verify: `psql -U ealedgers -d ealedgers -c '\dt'` shows the 14 tenant-scoped tables; `curl -sI https://ealedgers.com/admin/` returns 302; admin login works.

### Step 4 — Add `app-ci.yml` (CI gate)

What changes: New workflow `.github/workflows/app-ci.yml` triggered on `pull_request` and `push` to `main`. Single job: install requirements, `python manage.py check`, `python manage.py test accounting`. Block merges via branch protection on `main`.

Verify: Open a throwaway PR — CI run appears and passes. Push a deliberately-broken test — CI run fails and the PR shows a red gate.

### Step 5 — Co-tenant-safe `vps-bootstrap.sh` for ealedgers

What changes: Add `scripts/vps-bootstrap.sh`, modelled on apiworkflow's. Inspect-only behaviour on first run: enumerate existing vhosts under Apache, list other apps in `/opt`, refuse to touch anything it didn't create. Then idempotently install Python 3.13, virtualenv, the gunicorn systemd unit template, Apache vhost template (NOT certbot — that stays manual), local Postgres role/db.

Verify: Run on a throwaway VPS. Confirm `ap.cm-ea.com`, `bms.cm-ea.com`, `cmipaportal.com` vhosts (if present) untouched. Confirm `/opt/ealedgers/{app,venv,logs}` and the `ealedgers` user created.

### Step 6 — Capistrano-style release layout

What changes: Migrate `/opt/ealedgers/app/` to `/opt/ealedgers/releases/<id>/` + `/opt/ealedgers/current/` symlink + `/opt/ealedgers/shared/{.env,logs/,staticfiles/?}`. Update the gunicorn systemd unit's `WorkingDirectory` to `/opt/ealedgers/current`. One-time migration on the box, then commit the new layout to scripts.

Verify: `ls -la /opt/ealedgers/current` shows a symlink. Service still running. `systemctl status ealedgers` clean. Old `/opt/ealedgers/app/` removed last.

### Step 7 — Artefact-based `deploy.yml`

What changes: Replace the current `deploy.yml` (git-pull on host) with a two-job build-then-deploy pipeline. Job 1 (build, on runner): `pip install -t build/lib -r requirements.txt`, `python manage.py collectstatic --noinput`, tarball `build/ + manage.py + ea_accounting/ + accounting/ + staticfiles/ + scripts/vps-deploy.sh + requirements.txt`. Job 2 (deploy): scp tarball → new `releases/<id>/` → `pip install` → `migrate` → atomic symlink swap → `systemctl restart` → smoke check → keep last 5, prune older. Auto-rollback on smoke-check failure.

Verify: Trigger a deploy via `workflow_dispatch`. `ls /opt/ealedgers/releases/` shows new dir. Symlink swap atomic. Old release retained for rollback. Deliberately break a migration in a feature branch — deploy fails, rollback restores previous symlink, service stays up.

### Step 8 — Adopt plan-doc cadence

What changes: Add `docs/PROJECT_WBS.md` (running task list per phase with ✅/🟡/⬜ totals) and a `## Risk` section to every PR template. Optional: slice-numbered commit subjects when one feature spans multiple PRs (`Phase 2.1: invoices (slice 1/3)`).

Verify: Next merged PR uses the new shape. WBS document updates in the same PR that lands the feature.
